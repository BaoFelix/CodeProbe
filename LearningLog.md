## 一、5 条"大师级"思维模型(本段最值钱的部分)

### 模型 1:测试 fixture 是审讯员,不是验证员

**症状**:`test_src/` 上 100% 完美的引擎,在真实的 spdlog 上**关系解析率 4%**,top orchestrator 全是垃圾。换到 OCCT 又挖出 4 个新 bug。

**根因**:`test_src` 太干净——所有类大写命名、零别名、零导出宏、零第三方、零模板基类。**这些假设在它身上全成立,在真实代码里全不成立**。

**教训**:
- 玩具 fixture 只能证明"代码不会崩",不能证明"代码会对"。
- 真正的验证一定要**在风格各异的真实项目上跑**。每个项目都会暴露**不同**的隐藏假设。
- 这一段我们跑了 2 个项目(spdlog 应用型 / OCCT 分类库型)就挖出 **8 个 bug**。如果只跑 test_src,这些 bug 全会埋到生产环境。

**通用规则**:写完一个分析器,**至少在 3 个风格相异的真实项目上跑**,才能开始相信它。

---

### 模型 2:同一个工具,不同 scope,不同判断

**症状**:我们先猛批"正则解析代码不行",转头又在新代码里用了一堆正则。

**为什么不矛盾**:
- 正则**解析整份 C++ 源码** → 错的,语法太复杂,补丁补不完。
- 正则**解析"tree-sitter 已经抽干净的类型字符串"** → 对的,这是一个简单的子语言,正则刚好。

**教训**:
- 工具好不好用,不是固定的标签,看**你用它处理多大的东西**。
- 永远问:"这个工具的能力边界在哪?我现在喂给它的东西在边界内还是外?"
- 同理:Clang 是更准的解析器,但要求代码能编译——**在"扫半成品代码"这个场景下它的强项变成致命弱点**。

**通用规则**:选工具前先把"输入规模 + 输入特征 + 容错要求"写下来,再看哪个工具的能力曲线最匹配。

---

### 模型 3:留 NULL 比猜更聪明

**症状**:跨文件解析时遇到歧义短名(两个 namespace 都有 `Plot`),我们选择**留 target_qname=NULL,不猜**。

**为什么**:
- 猜错一次 → 依赖图错一片 → orchestrator 算分错 → LLM 拿到污染的输入 → 全链路错。
- 留 NULL → LLM/下游知道"这边没解析出来",能保守判断。
- **错的数据比缺失的数据更危险**,因为你不知道它错了。

**教训**:
- 在不确定的地方,**显式标记 unknown**,而不是用"最可能的答案"填上。
- 工程上这叫 *fail closed*(失败时关闭),静态分析、数据库、安全系统、网络协议都是这个原则。
- 反例:用大写首字母判类名,看似"差不多",到 spdlog 上全军覆没——**软性约定不是规则,是定时炸弹**。

**通用规则**:每个不确定的字段都该有"显式 unknown"选项,而不是默认值伪装成确定。

---

### 模型 4:数据模型决定能问的问题

**症状**:旧 schema 里"方法只是 class 表的一个 method_count 整数"。这意味着我们**永远没法问**:"Workshop.Repair() 调用了 Engine 的哪些方法?"——因为方法不是一等公民,根本不在表里。

**为什么**:
- Schema 决定能问什么 → 决定能算什么 → 决定能让 LLM 看到什么。
- 新模型把方法、字段升级为实体 → 立刻能查"X 类内部有什么",支持将来加方法级 calls 边。
- `parent_qname` 一列 → 解锁"包含关系"的所有查询(给我 X 内部的全部子节点),旧 schema 完全做不到。

**教训**:
- 设计 schema 时不要只问"现在需要存什么",要问"**将来想回答什么问题?这个 schema 能回答吗?**"
- 错误的 schema 是隐藏的天花板。代码再聪明也突破不了它。
- 好的 schema 让"将来想做"变成"将来加点代码就行"。坏的 schema 让"将来想做"变成"推倒重来"。

**通用规则**:Schema 评审用"未来 5 个问题"测试:列出未来可能想回答的 5 个问题,看 schema 能不能回答(或便宜地扩展去回答)。

### 模型 5:折叠/聚合是"看的方式",不是"事实"

**症状**:同一个调用图,
- 折到顶 → spdlog 看起来漂亮、OCCT 看起来扁平
- 只折叶子 → spdlog 略复杂、OCCT 分类层次完整

**为什么**:**数据是一样的,折叠方式不同 → 同一份数据呈现出不同的"事实"**。

**教训**:
- 任何"压缩信息让人看"的步骤,本质都是**选了一个角度**。
- 同一份数据应该支持**多种压缩方式**(我们做了 3 档模式),让用户挑视角。
- 别把"我选的视角"当成"代码的事实"——这是初学者最容易犯的错。

**通用规则**:任何聚合/折叠/排序/截断,都应该问"我为什么选这个角度?换一个角度看会是什么样?"。把这个选择**显式化、可配置**,不要藏在代码里。

## 二、8 个真实 bug 全清单(可迁移到任何静态分析项目)

每个 bug 都是"test_src 玩具暴露不了 + 真实代码必然遇到"的类型。记下来,你下次写代码分析器时直接拿去当检查表。

| # | bug | 表面症状 | 真实根因 | 教训 |
|---|---|---|---|---|
| **A** | 类名大写假设 | `vector<sink_ptr>` 抽不到关系 | 正则写了 `[A-Z]\w+` 要求大写开头 | 不要拿"命名风格"当判别规则——命名风格因项目而异 |
| **B** | typedef / using 别名 | `logger→sink` 这条最关键的边连不上 | spdlog 用 `using sink_ptr = shared_ptr<sink>` 包了一层 | 真实 C++ 大量用别名,别名链必须能展开 |
| **C** | 第三方目录污染 | top orchestrator 全是 fmt 库的类 | spdlog 内嵌了整个 fmt 库 | 默认排除 vendored/bundled/external,不让第三方污染统计 |
| **D** | 类级导出宏 | `class SPDLOG_API logger` 抽出"SPDLOG_API"为类名 | tree-sitter 不展开宏 | 解析前用等长空格抹掉宏(保持行号) |
| **E** | 前向声明幽灵 | OCCT 抽出一堆只有名字没有内容的"类" | `class gp_Trsf;` 被当成定义 | 没有 body 的 class 节点必须跳过 |
| **F** | 方法级导出宏 | OCCT 抽出"Standard_EXPORT"为返回类型 | 同 D 但在方法位置 | 通用化:任何 `*_EXPORT/*_API` token 全局抹掉 |
| **G** | 智能指针 handle | `occ::handle<X>` 没归到 associates | 我们只认 shared_ptr,不认 OCCT/Qt/boost 的等价物 | 别用"标准库专属"的规则,要支持等价语义 |
| **+** | 纯虚 `=0` 三义性 | 21 个普通类被误判 abstract | `void f(int x=0)` 默认参数、`{x=0;}` 方法体内容、`)=0;` 纯虚——长得像 | 一个语法片段在不同位置可能完全不同语义 |

**附加教训(tree-sitter API 坑)**:
- `captures()` 返回各 capture 名分组,但**分组之间顺序不一致**——zip 它们会张冠李戴。用 `matches()` 按"一次匹配"打包返回才安全。
- C++ 的"方法"在 AST 里有至少**三种形态**:`field_declaration`(普通声明)、`declaration`(构造/析构)、`function_definition`(内联实现)。query 三种都要写。
- 模板基类是 `template_type` 节点,不是 `type_identifier`。继承 query 不写 template_type 就漏掉所有 `class X : public Y<T>`。

## 三、决策日记:我们为什么选 A 不选 B

这一段每个十字路口,我都让你拍板了。把决策原因记下来,下次类似十字路口直接复用判断。

### 决策 1:用 tree-sitter 而不是 Clang 或正则

**选 tree-sitter** 因为:
- 跨语言(一套 query 框架,换 grammar 就行)
- 容错(代码写一半也能解析)
- 快(IDE 实时高亮就是它)

**弃 Clang** 因为:它要代码能编译,我们要扫半成品,致命冲突。
**弃正则** 因为:它不懂语法,补丁补不完。

**通用原则**:选解析器时把"是否容错"放在"是否精确"前面,如果你扫的不是单一项目的完整 build。

### 决策 2:实体-关系模型,不是属性表

**选了实体-关系** 因为方法/字段成为一等公民,可扩展性强(将来加 calls 边方法可作为关系端点)。
**弃了** 旧的"class 表 + dependencies 表"因为方法/字段被压成整数,挖不出来。

**通用原则**:抽象级别低一档,可扩展性高一档。设计 schema 默认走"实体化所有可命名的东西",除非有性能反证。

### 决策 3:parent_qname 字符串,不是 parent_id 外键

**选了字符串** 因为批量插入更简单(不用先插父查 id 再插子)、调试可读。
**弃了** 外键 因为我们每次重建图,不需要 SQL 层完整性保证。

**通用原则**:在"你完全控制数据生命周期"的场景里,关系完整性约束的价值降低,可读性的价值升高。

### 决策 4:合并 calls 进 depends(YAGNI)

**选了合并** 因为:
- 你的真实目标(orchestrator + LLM 上下文)不需要方法级粒度
- body call 抽取在 C++ 里又难又脆(虚函数、模板、重载)
- 真要做可以用 `attrs.via = 'body_call'` 区分,模型不变

**弃了** 分开因为这是为想象中的未来需求买单(YAGNI)。

**通用原则**:有人(包括自己)鼓吹"为将来留扩展"时,先问"将来要做这个吗?如果做,现在的设计能扩展吗?"。如果两个答案都是"也许",**砍掉它**。

### 决策 5:body call 不做,先看现有数据够不够

**选了先验证** 因为:用 Vehicle.cxx 真实数据看了一眼——方法体调用的类,**100% 都在字段关系里已经抽到了**。新增方法级 calls 的 ROI = 接近 0。
**弃了** 凭感觉先做。

**通用原则**:加复杂度前,**用真实数据证明它带来新信息**。"似乎更全面"不是理由。

### 决策 6:折叠默认 leaves,不是 all 也不是 none

**选了 leaves** 因为:
- 对应用型代码(spdlog)叶子多,折叠效果明显
- 对分类库(OCCT)叶子被收掉,中间分类层(Curve/Surface/Conic)保留
- **一套规则自适应两种风格**,不需要让用户挑

**弃了 all** 因为压平了 OCCT 的分类学。
**弃了 none** 因为 spdlog 直接 98 个根、用户开箱看到的是噪音。

**通用原则**:有默认值的可配参数,默认值应该是"在最常见两类输入上都不出大错"的那档,不是"对某一类最优"的那档。

## 四、我犯过的错(自我修正记录)

诚实记录,免得下次再犯。

### 错 1:静默改设计

Phase 1 我提议 schema 用 `parent_id`,代码里悄悄改成 `parent_qname`,**没告诉你**。你后来发现"这个 parent_qname 是什么?"质问我。

**教训**:任何偏离已沟通方案的改动都必须**告诉对方 + 给对方拒绝的机会**,哪怕你觉得改得"显然更对"。沉默改设计 = 慢性破坏信任。

### 错 2:用学院派 jargon 解释

讲 parent_qname 价值时我说"包含关系"、"作用域可见性"、"递归结构"——你卡住说"看不懂能用大白话讲吗"。

**教训**:技术正确 ≠ 沟通有效。**默认用大白话 + 具体例子**,需要术语时显式定义。"我懂"不意味着"你懂"。

### 错 3:把"应该测试"挂在嘴边,但没真测

我从 Phase 1 一路说"在真实项目验证才扎实",但每一步都拿 test_src 当"通过测试"。直到你主动说"换真实项目压测",我才动手——结果一次性挖出 8 个 bug。

**教训**:"我知道应该 X" ≠ "我会主动 X"。设计 checkpoint(每完成一大块就必须真实数据验证一次),不要靠自觉。

## 五、下次还能怎么进化

留给"将来再来一段"的我:

1. **跑第三个真实项目** — spdlog(应用型) + OCCT(分类库型)我们覆盖了两类。下一个找:**多继承/CRTP 重的(Eigen / Boost)**、**有大量 free function 的 C 风格库(curl)**、**有大量宏的代码(linux 内核 / qemu)**。每个会暴露新 bug 模式。

2. **方法级 calls 的可行子集** — 真要加时,只解析两种(占 90% 调用):
   - `成员字段.方法()` ← 字段类型我们已经知道
   - `参数.方法()` ← 参数类型我们已经知道
   其他(局部变量、工厂返回值)直接跳过。产出 `depends` 边 + `attrs.via='body_call'` + `caller_method` 属性。

3. **接口判定升级** — 当前规则:"全纯虚 + 无字段 = interface;有纯虚 = abstract"。可扩展:
   - 看是否被用作 `Handle<X>` / `shared_ptr<X>` 字段的多个实现 → 即使没纯虚也是事实上的接口
   - 看 abstract 类是否有 `factory_method()` 之类的工厂签名

4. **打分系数自适应** — 现在 `+0.5×reach -0.8×in` 是写死的。可以在每个项目上跑一遍找出"最像 orchestrator 的 3 个类"(人工标注),然后微调系数让排序匹配——一个最小 ML 步骤。

5. **可视化** — 现在只有 ASCII。`networkx` → `pydot` → SVG 几十行就能出。但要等真接入 pipeline 之后做。
